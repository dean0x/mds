### MEDIUM

**Mutable frozen-object bypass via spread** - `packages/mds/src/backend/wasm.ts:106,117`
**Confidence**: 82%
- Problem: `DEFAULT_COMPILE_OPTS` is declared with `Object.freeze()` containing `modules: {} as Record<string, string>`. The frozen empty object is shared across all compile/check calls. When `vars` is present, the spread `{ ...DEFAULT_COMPILE_OPTS, ...vars }` creates a new object (safe). But when `vars` is `undefined`, the frozen `DEFAULT_COMPILE_OPTS` reference is passed directly to `wasm.compile()`. If the WASM bridge mutates the options object (e.g., adding properties to `modules`), the frozen object will silently reject writes in non-strict mode or throw in strict mode. This is defensive but could mask bugs. More importantly, the `modules` sub-object `{}` inside the frozen object is only shallow-frozen — `Object.freeze` does not deep-freeze. If the WASM bridge writes `opts.modules['key'] = value`, that mutation would persist across calls, leaking module content between unrelated compilations.
- Impact: Cross-request data leakage if WASM bridge mutates the `modules` property of the options object. Severity depends on whether the WASM bridge is well-behaved. Since the WASM bridge is auto-generated by wasm-pack, mutation is unlikely but not guaranteed.
- Fix: Deep-freeze or create a fresh object per call for the no-vars path:
  ```typescript
  const DEFAULT_OPTS = { filename: 'input.mds' } as const;
  // In compile/check:
  return wasm.compile(source, vars !== undefined
    ? { ...DEFAULT_OPTS, modules: {}, ...vars }
    : { ...DEFAULT_OPTS, modules: {} });
  ```

## Pre-existing Issues (Not Blocking)

### MEDIUM

**Init options silently ignored on second call** - `packages/mds/src/browser.ts:37-50`
**Confidence**: 85%
- Problem: `init(options)` caches the promise from the first call. If a second caller invokes `init({ wasmUrl: '/custom.wasm' })` after an earlier `init()` already started or completed, the `options` argument is silently discarded — the backend uses whichever URL the first caller provided (or the default). This is pre-existing behavior (the old code had the same pattern via `doInit`), but the new code makes it slightly more subtle because `createWasmBackend(options)` is only called on the first invocation.
- Impact: Low in practice — documented as idempotent. But a caller passing conflicting `wasmUrl` would get no indication their configuration was ignored. Not a vulnerability per se, but a usability/safety concern in multi-entry-point applications.

### LOW

**Error message includes absolute filesystem path** - `packages/mds/src/util/module-scanner.ts:126-127,139,172-174,179-181`
**Confidence**: 80%
- Problem: Security-related error messages include absolute paths like `security: import path escapes project root: /Users/foo/bar is outside /Users/foo`. In server-side usage, these paths could leak internal directory structure to callers if errors propagate to HTTP responses.
- Impact: Information disclosure of server filesystem layout. The Rust compiler already has this pattern, and this is a library (not a server), so the risk is indirect — it depends on how callers handle errors.
- Note: This is pre-existing code, not introduced in this diff. Informational only.

## Suggestions (Lower Confidence)

- **Aggregate size race with parallel scans** - `packages/mds/src/util/module-scanner.ts:188` (Confidence: 65%) — The `aggregateSize` variable is mutated non-atomically from concurrent `Promise.all` branches. In JavaScript's single-threaded event loop this is safe because `+=` on a number is synchronous between awaits, but the pattern is fragile if future refactoring introduces async operations between the read and write of `aggregateSize`. Consider documenting this single-threaded invariant.

- **WASM module candidates use relative URL resolution** - `packages/mds/src/backend/wasm.ts:69` (Confidence: 62%) — The WASM candidate path is constructed with `new URL('../../../../crates/mds-wasm/pkg/mds_wasm.js', import.meta.url).pathname`. The `../../../../` traversal is hardcoded for the workspace layout. A packaging or deployment change could cause this to resolve outside the expected directory. The `createRequire` + `require()` call mitigates arbitrary code execution risk since it loads a JS module, not arbitrary data, but the path construction is brittle.
